INTRODUCTION

bi(OS)® is a Hyper-converged data managementTM platform delivering unprecedented agility to API, AI, and BI builders in a unified way.  It converges all aspects of Data Management, DataOps, and ModelOps into a single platform.  With bi(OS), the complete lifecycle of building data apps is reduced from months to weeks. This includes adding varied data sources, deriving real-time insights, and deploying to production.

CONSTRUCTS

bi(OS) is modeled using constructs from the human sensory system, and there are two constructs to deal with - signals and contexts.

Signals

Signals represent real-world events/observations.  Once an observation is recorded it’s immutable.  Examples of signals are the current price of a stock, clicks in cyberspace, thermostat reading from an IoT sensor, etc.

Signals are fit for data sources that are -

  • Voluminous - from hundreds to millions of events/sec
  • Semi-structured - Collection of attributes modeled as key/value pairs
  • Noisy - Upstream sources can introduce errors, change the schema, etc.

Contexts

Contexts represent the state of a system that can change/evolve. Examples of contexts are mapping of SKU to price, IP to GEO, customer to the probability of churn, etc.

Contexts are fit for data sources that are -

  • Relatively static - While entire context can contain millions of records, only a small percentage change regularly
  • Semi-structured - Collection of attributes modeled as key/value pairs, with one attribute being the key, e.g. SKU, IP, CustomerId, etc.
  • Agile - Contexts can be added, removed, changed, etc.

OPERATIONS

All of the below operations are configurable within bi(OS) via the UI or the SDK.  And all of the below operations can be changed while live ingestion is going on. bi(OS) encourages fearlessness in data tinkering.  See unique capabilities here.

Validation

bi(OS) can perform attribute-level validation during inserts/upserts.  

There are two types of validations performed -

  • Missing values - If an attribute value is missing -
  • Reject the insert/upsert
  • Fill in default values
  • Allowed Values - Currently, this validation is executed for text attribute types only.

Missing value validation can be set at an attribute-level or globally for the signal or context.  If there is a conflict, the attribute-level policy takes precedence.

Enrichments

Contexts can enrich signals during inserts.  A particular attribute within a context is defined as its primaryKey.  When a signal is being inserted, an attribute within the signal (known as a foreignKey) is used to lookup against this primaryKey field within the context.

There are two possible outcomes -

  • If lookup matches in value, the enrichment action proceeds.  This enrichment appends user-selected attributes from the context to the signal.  
  • To avoid name conflicts among attributes, an option to provide an alias is available.
  • If the lookup operation fails there are two options -
  • Reject the insert - signal insertion is rejected.
  • Use fill-in values - Use values that can be filled-in for enrichments

Note - The option to use fill-in or not has to be enabled for each enrichment and the values to use for fill-ins have to be provided for each attribute within a context that was going to be appended to the signal.

Enrichments have the following three powerful capabilities -

  • Enrichments are executed before the signal data is stored; in other words, they are truly real-time.
  • It’s possible to do one (signal) to many (context) enrichments[1].
  • It’s possible to nest enrichments[2], i.e., attributes appended as part of any particular enrichment can be used as a foreignKey to perform future enrichments.

Feature Computation

Features are computed after the data is stored.  The default interval at which this computation happens is set to five minutes[3] and change be changed by the user.

There are two types of features -

  • Index features - These features are used to analyze a signal across multiple (possibly hierarchical) dimensions, e.g., Country, State, City, OR Product Category, Product, etc.[4]
  • Aggregate features - These features represent essential aggregates on numerical attributes within a signal.  If enabled by the user, bi(OS) auto-computes min, max, sum, and count for any numerical attribute within the signal[5].  These features serve alerting, anomaly detection, and insights.

Alerting

If configured, aggregate features are passed through a rule engine to generate alerts to user-defined web-hooks[6].  The expression language for alerts is defined in the SDK documentation and a simplified interface is also available via the UI.  Users can also configure streaming of aggregate features by setting an alert condition which always evaluates to true.

Anomaly Detection

If enabled, aggregate features computed every interval are passed through anomaly detection logic and compared with the last ten samples.  If the value of any aggregate feature is different by three-sigma (3σ), a special alert - anomaly detected - is generated.

Insights

Aggregate features are used to power real-time BI via insights on the left navigation bar.

UNIQUE CAPABILITIES

bi(OS) encourages tinkering with real-time data.  

Changes that can be done while ingestion is live are -

  • Attributes - Ability to add, remove, change validation, data-type, for attributes within signals or contexts
  • Signals/Contexts - Ability to add, remove, signals/contexts on a production system.
  • Enrichments - Ability to add, remove, update enrichments
  • Features - Ability to add, remove, update features, and let them auto-compute for live and already ingested data in a deterministic manner.

These powerful capabilities encourage tinkering by data practitioners.  Some things to note -

  • Every change in the system is versioned, and isQLTM select allows querying of data across changes.  This capability requires default values to be provided whenever changes are provisioned.
  • Anytime an attribute is added or removed, it may require minor retrofitting to the ingestion code.
  • The following changes don’t require any code changes -
  • Adding/removing an enrichment to an existing signal
  • Changing validation rules for existing attributes
  • Changing data-types of existing attributes without changing the total number or order of attributes
  • Adding, removing or changing a feature

ENTERPRISE READINESS

Security

bi(OS) supports basic authentication using passwords and role-based access control (RBAC) for authorization (as defined in the persona section above).  Further, every interaction with bi(OS) and within bi(OS) is encrypted over SSL.

Resiliency

bi(OS) stores signals and contexts across three distinct machines and availability-zones (on the public cloud).  It can be deployed in geo-redundant environments across multiple data centers if needed.

Deployment Options

bi(OS) runs on commodity servers with commodity storage.  Both local and network-attached storage options have been validated to work adequately with bi(OS).  Local-attached storage provides the best cost-performance benefit.

Name

contextName / signalName

A case insensitive alphanumeric string less than 40 characters that uniquely identifies a signal or context.  It is highly recommended to use camelCase convention for naming signals and contexts.

Policies

Global policies that govern validations for all attributes within a signal or context.  These can be overridden for a specific attribute if required.

missingAttributePolicy

Governs the behavior for all attributes within a context or signal if their values are missing during insert/upsert. The following actions are possible -

  • Reject - If the value is missing insert/upsert will return an error
  • StoreDefaultValue - Use the default specified at the attribute-level.  Specifying a default at attribute-level is mandatory if this is set[7]

Attribute Configuration

An arbitrary number of attributes can be defined as part of context/signal configuration

    {

      "attributeName": "gender",

      "type": "string",

      "allowedValues": ["MALE", "FEMALE", "UNKNOWN"],

      "missingAttributePolicy": "StoreDefaultValue",

      "default": "UNKNOWN"

    }

attributeName

Name of the attribute

type

The following data types are supported

  • string
  • integer (int64)
  • decimal (double, float)

missingAttributePolicy

Used to override the setting at the global (signal or context) level.

default

The default value to use if an attribute is empty and missingAttributePolicy results in StoreDefaultValue[8]

allowedValues

For strings which values are allowed

  • A list of strings (enumeration) can be supplied
  • If missingAttributePolicy is set to store a default value, the default value should be part of the list

missingLookupPolicy

Governs the behavior if the lookup into a context results in an empty match.

This policy can be applied at global level (i.e. for all enrichments) and can be overridden for individual enrichments.

Allowed actions are -

  • Reject - insert operation is rejected
  • StoreFillInValue - Use the configured fillIn value as a substitute

enrichmentName

Name for the enrichment which is unique for a signal

foreignKeys

The keys on which the enrichment from context is performed. Currently, only a single attribute is supported

contextName

Name of the context with which the enrichment is configured

attributeName

The context attribute(s) that will be enriched with the signal[11]

as

The name to use for the enriched attribute (within the signal)[12]

fillIn

Value to be used in case of lookup into a context results in an empty match and missingLookupPolicy results in StoreFillInValue

Note: Enrichments can be chained in the sequence in which they are specified in configuration.  The default setting for bi(OS) is to allow a maximum of two levels of enriched attribute chaining[13].

          "reportedDeaths"

        ],

        "metrics": [

          "count()",

          "sum(reportedCases)",

          "sum(reportedDeaths)"

        ],

        "rollupInterval": 300000,

         "alerts": [

           {

             "alertName": "alertWhenDeathsCross1000InADay",

             "condition": "sum(reportedDeaths) > 1000",

             "webhookUrl": "https://webhook.site/99743393-3a47-473f-8676-319e8c5d9422"

           }

         ]

      }

    ]

featureName

Name of the feature. Names are case-insensitive. Alpha-numeric and special  characters are allowed. They must be unique for a signal.

dimensions

Set of attributes that act as dimensions for the feature.  These dimensions are used during querying to optimize return results.

attributes

Set of attributes for which metrics are computed

metrics

The following metrics are supported -

  • count
  • min
  • max
  • sum

Specific combination of metrics with attributes need to be configured ex: sum(reportedCases), max(reportedDeaths)

rollupInterval

In milliseconds[14]. Periodic interval for rolling up ingested data

alertName

Name of the alert.  Names are case-insensitive. Alpha-numeric and special  characters are allowed. Alert names must be unique for a signal.

condition

The condition that will be evaluated so that an alert can be sent.

webhookUrl

The url of the webhook to which the generated alert will be posted

Rules

  • Conditions should be in parenthesis

SELECT FROM Signal

The select method provides flexibility to perform simple record fetches to complex summaries with tumbling and sliding windows.  All select operations return a SelectResponse object which consists of DataWindows, which consists of records.

SelectResponse - Python

The bios.models.SelectResponse consists of

  • data_windows
  • records

SelectResponse - Java

The io.isima.bios.models.SelectResponse consists of

  • ISqlResponseType (always set to ISqlResponseType.SELECT)
  • List<DataWindow>

Data windows

The data returned from a select query can be partitioned into multiple windows based on the query. For example, if an hourly summary is requested for the last 12 hours, then 12 windows would be returned each containing a summary per hour.  

bi(OS) provides two types of windowing capability as described below.

Tumbling window

These windows have fixed size (time interval) and do not  overlap. Only window size and overall duration are required for calculating the metrics.

Sliding window

These windows have fixed size (time interval) and overall duration. But, a sliding interval (some fraction of the window size) determines, how far back would the system go for calculating the metrics data.

Duration

Duration is always specified as two integer values -

  • start        -  milliseconds offset from UTC
  • delta        -  offset (in milliseconds) from the start (positive or negative), depending on whether data ahead or behind the start is to be fetched

Records

These contain the records returned by the system for a given window. If the SELECT query does not request a tumbling or sliding window, all the data is returned in a single window.

Refer below for printing all the records within all data windows.

Python syntax

for window in response.get_data_windows():

        print('Deaths  -> ' + str(record.get('reportedDeaths')))
